  • How can you plan and implement database upgrades?

    Perform a "Smoke Test" using the User Interface to test the database is working in a stable way. Sometimes due to sensitivity of the data, you might night be the one who does the "Smoke Test", however whoever the person is, make sure they are present before upgrading or you can't validate the upgrade is working until users call and complain of an outage (which you don't want to happen).

  • How can you plan and implement database upgrades?

    If possible, consider an upgrade plan where the Database schema is versioned based on packages or namespace scope. Why? Because you will want to carry the old way of working for at least 2 versions before dropping a column, renaming a column, or adding significant data changes so some functionality can be rolled back without losing any data. Database Snapshots are not an effective way because they rollback the data and the schema changes and users will get annoyed if their data is deleted.

    Dan Swaney contributed 1 year ago Upvote

  • How can you plan and implement database upgrades?

    You can often test the migration in a Docker Container of a clean baseline of the Database Version. Export a data dump of the Production Database to use for testing. Load the Exported Data Dump into the Docker Container Database. Perform a test migration or upgrade to see what happens. Perform a test of access to the data as a "Smoke Test" from a User Interface setup to use the Database. Create a Package Deliverable Folder with all the needed data files, scripts, and documentation on the upgrade process and rollback. As with any migration, there will be risk. Think of ways to mitigate the risk and minimize downtime if something goes wrong so as to limit an outage for the user.

    Dan Swaney contributed 1 year ago Upvote

  • How can you plan and implement database upgrades?

    - When selecting an Upgrade Method to a Database, understand first what is being upgraded. Is this a Schema Change Upgrade? Is this a Database Software Upgrade? Is this a Product Migration Upgrade? Can you do a cutover upgrade to minimize downtime for users to near zero? What if the Upgrade Process fails? Some application products have tools to migrate from one product to another so you don't have to do the migration coding. Others use tools to massage the data before importing. Make sure you understand what specifically is being migrated, how long it takes, and whether there will be downtime required. And make sure you back everything up first so you can restore it if something goes wrong. This can be done in advance of the Upgrade.

    Dan Swaney contributed 1 year ago Upvote

  • How can you plan and implement database upgrades?

    - Some vendors, like PostgreSQL, have live webcast presentations to explain what benefits you will get from upgrading. For example, PostgreSQL released a new version last year which eliminated the UDP sockets sending metrics out whether someone was listening or not. Now they changed it to where you have to explicitly connect to it internally to collect the metrics and relay them which is more secure and less of a flag for security scanners to flag as a vulnerability. There should always be a reason for upgrading vs upgrading just to do so. Anytime an upgrade is done, there is added risk to break what is working.

    Web accessible content stored in obscure, unpopular or obsolete formats represents a signiﬁcant problem for digital preservation. The ﬁle formats that encode web content represent the implicit and explicit choices of web site maintainers at a particular point in time. Older ﬁle formats that have fallen out of favor are obviously a problem, but so are new ﬁle formats that have not yet been fully supported by browsers. Often browsers use plug-in software for displaying old and new formats, but plug-ins can be diﬃcult to ﬁnd, install and replicate across all environments that one may use. We introduce Grace, an http proxy server that transparently converts browser-incompatible and obsolete web content into web content that a browser is able to display without the use of plug-ins. Grace is conﬁgurable on a per user basis and can be expanded to provide an array of conversion services. We illustrate how the Grace prototype transforms several image formats (XBM, PNG with various alpha channels, and JPEG 2000) so they are viewable in Internet Explorer.
